# Smart Parking SPA — UI glossary и tone-of-voice guide

Дата: 2026-04-21  
Основание: `docs/FRONTEND_FINAL_AUDIT_2026-04-20.md`

## 1) Зачем нужен этот документ

Этот гайд фиксирует единый стиль и словарь пользовательских текстов во frontend Smart Parking SPA, чтобы:
- убрать смешение RU/EN и «технического» copy в интерфейсе;
- сделать тексты короткими, понятными и единообразными;
- упростить ревью всех будущих frontend-задач с user-facing copy.

## 2) Tone of voice

Целевой стиль интерфейса: **деловой, понятный, современный**.

Правила:
- Пишем короткими фразами, без канцелярита.
- Обращаемся к пользователю на «вы» в нейтральной форме.
- Глаголы — в активной форме: «Сохранить», «Подтвердить», «Отменить».
- В одном сообщении — одна мысль.
- Избегаем жаргона и внутренних терминов API.

## 3) Что нашли в текущем copy (по аудиту и коду)

### 3.1 Конфликтующие термины
- `Неявка` / `no_show`.
- `Свободно` / `Забронировано` vs целевая пара статусов занятости.
- `Журнал аудита` / `логи аудита` / «история изменений» (смешение сущностей и уровня детализации).

### 3.2 Дублирующие формулировки
- Разные варианты для одного действия: «Продолжить», «Подтвердить», «Изменить роль» без единого шаблона подтверждения.
- Разные названия для operational copy о бронированиях: «поток», «жизненный цикл», «действия по статусу».

### 3.3 Английские заимствования в user-facing тексте
- `rules`, `full replace semantics`, `Access mode`, `Allowed roles`, `Working hours`, `Schedule exceptions`, `No/Yes`, `Step`, `Advance`, `open/close time`.

### 3.4 Слишком технические сообщения
- Сообщения с API-деталями (`PUT`, endpoint-специфика) без пользовательской пользы.
- Вывод служебных полей и кодов без расшифровки в тексте.

## 4) Базовые правила copy

## 4.1 Короткие CTA
- 1–3 слова.
- Глагол в начале.
- Примеры: «Сохранить», «Подтвердить», «Отменить», «Открыть детали».

Не рекомендуется:
- «Выполнить операцию», «Произвести изменение».

## 4.2 Названия действий
- Для статусов бронирования используем: «Подтвердить», «Отменить», «Отметить прибытие», «Отметить выезд», «Отметить неявку».
- Для управления данными: «Создать», «Изменить», «Удалить», «Сбросить фильтры».

## 4.3 Ошибки
Шаблон:
1) что произошло, 2) что сделать дальше.

Пример:
- «Не удалось загрузить уведомления. Обновите страницу и повторите попытку.»

## 4.4 Empty / loading states
- Empty state: причина + следующее действие.
- Loading: нейтральная короткая форма («Загрузка данных…»), без технических деталей.

## 4.5 Подтверждающие диалоги
- Заголовок: действие в повелительной форме или вопрос.
- Описание: одно предложение о последствии.
- Кнопки: `Отменить` + целевое действие (`Удалить`, `Изменить роль`, `Завершить`).

## 5) Единый словарь Smart Parking

| Сущность | Используем | Не используем |
|---|---|---|
| booking | **бронирование** | booking |
| parking | **парковка** | lot (без контекста) |
| parking spot | **парковочное место** | spot |
| spot status | **доступно / занято / заблокировано** | свободно / забронировано (как primary-пару статусов занятости) |
| no-show | **неявка** (статус), **неявки** (метрика) | no-show, неявка |
| completed | **завершено** | complete |
| cancelled | **отменено** | canceled/cancel |
| confirmation | **подтверждение** | confirm step |
| management | **управление** | management |
| analytics | **аналитика** | analytics |
| notifications | **уведомления** | notifications |
| rules | **правила** | rules |
| audit/changes | **журнал изменений** (для сущности audit log), **история изменений** (для timeline в карточке сущности) | log stream, old_values/new_values в явном виде в пользовательских заголовках |

## 6) Запрещенные и нежелательные формулировки

Запрещено в user-facing тексте:
- Смешение RU/EN в одной фразе.
- Нерасшифрованные служебные коды (`no_show`, `old_values`, `assignment_metadata`) как подписи UI.

Нежелательно:
- Канцелярит: «выполнить осуществление», «произвести операцию».
- Перегруженные предложения длиннее 2 строк в типовом viewport.

## 7) Что обновлено в коде в рамках первого внедрения

- Выровнены статусы доступности парковочного места под словарь: `доступно / занято / заблокировано`.
- В аналитике по бронированиям унифицирован термин no-show: `Неявка` (статус) и `Неявки` (метрика).
- В редакторе правил парковки заменены англоязычные UI-строки и технические англицизмы на русские и более понятные формулировки.

## 8) Обязательные правила для всех следующих frontend-изменений

1. **Перед merge любого PR с UI-текстами** проверить строки по разделу «Единый словарь».  
2. **Новые статусы и CTA** добавлять только в канонической форме из этого гайда.  
3. **Любой новый empty/error/confirm текст** писать по шаблонам из раздела 4.  
4. **Не выводить сырой API-язык пользователю**; при необходимости — переводить в понятные термины.  
5. **При ревью** отклонять PR, где есть RU/EN mixing в user-facing copy без продуктового обоснования.

# Материалы из кода для написания дипломной работы

> Документ подготовлен на основе анализа backend (`app/`), frontend (`frontend/src/`), миграций (`alembic/versions/`), тестов (`tests/`) и существующей документации (`docs/`). При расхождениях приоритет отдавался коду.

## 1. Общая характеристика проекта

Кратко: проект реализует цифровую систему управления парковкой с ролями пользователей, управлением парковками/местами, бронированиями, контролем жизненного цикла брони, рекомендациями по выбору места, аналитикой, уведомлениями, журналом аудита и модулем ANPR (распознавание номерных знаков).

### Структура и компоненты

| Компонент | Где находится | Назначение | Что можно использовать в дипломе |
|---|---|---|---|
| Точка входа backend | `app/main.py` | Инициализация FastAPI и фонового планировщика sync статусов бронирований | Архитектурная схема и описание lifecycle-процессов |
| API v1 роутер | `app/api/v1/router.py` | Подключение endpoint-модулей (`auth`, `parking`, `bookings`, `recommendations`, `analytics` и др.) | Полный перечень функциональных подсистем |
| Endpoint-слой | `app/api/v1/endpoints/*.py` | HTTP API для всех доменных процессов | Раздел проектирования API |
| Сервисный слой | `app/services/*.py` | Бизнес-логика (бронирования, рекомендации, аналитика, уведомления, аудит, lifecycle) | Раздел реализации backend и интеллектуальности |
| Модели БД | `app/models/*.py` | SQLAlchemy-модели: пользователи, парковки, места, брони, зоны, авто, события доступа и т.д. | ER-диаграмма и раздел моделей данных |
| Pydantic-схемы | `app/schemas/*.py` | DTO запросов/ответов API | Формализация контрактов API |
| БД и сессии | `app/db/*.py` | Подключение БД и базовая ORM-конфигурация | Описание инфраструктуры данных |
| Миграции | `alembic/versions/*.py` | Эволюция схемы БД: роли, статусы, индексы, аудит, уведомления, правила | История проектных решений |
| Frontend SPA | `frontend/src/*` | UI для ролей, маршрутизация, формы, таблицы, аналитические экраны | Раздел UI/UX и интеграции |
| API-клиент frontend | `frontend/src/shared/api/*`, `frontend/src/entities/*/api.ts` | HTTP-клиент, токены, обработка ошибок, сущностные API-модули | Раздел интеграции FE-BE |
| Тесты | `tests/*.py` | Проверка auth, bookings, analytics, recommendations, audit, notifications, parking rules и т.д. | Раздел верификации и качества |
| Документация проекта | `docs/*.md` | Материалы по lifecycle, ANPR, аналитике, frontend mapping | Основа для текстов диплома + сопоставление с кодом |

### Технологии (по коду)
- Backend: FastAPI, SQLAlchemy (async), Pydantic Settings, Alembic.
- Auth: JWT (см. `app/core/security.py`, `app/services/auth.py`, `app/api/v1/endpoints/auth.py`).
- Frontend: React + TypeScript + Vite + React Router + MUI + React Query + Axios (`frontend/package.json`).
- Интеллектуальные/алгоритмические модули: рекомендации мест (`app/services/recommendations.py`), anomaly detection (`app/services/anomaly_detection.py`), forecast/analytics (`app/services/analytics.py` + `docs/occupancy_forecast.md`).
- ANPR: пайплайн и провайдеры (`app/services/anpr/*`, `app/services/plate_recognition_pipeline.py`).

### Что уже реализовано / ограничения
- Реализовано: роли, CRUD парковок и мест, бронирование + проверка пересечений, auto-assign, lifecycle статусов, рекомендации с explainability, уведомления, аудит, аналитика, управление пользователями админом, доступ/события въезда.
- Частично: “интеллектуальность” реализована в rule/scoring-подходе (не ML-first). Прогнозирование/аналитика есть, но требует демонстрационных данных и методической валидации.
- Ограничения: часть модулей зависит от корректного заполнения данных; ANPR Runoi опционален и требует тяжелых зависимостей (`requirements-anpr.txt`).

## 2. Материалы для ВВЕДЕНИЯ

### 2.1 Актуальность (факты из проекта)
- Есть цифровой учет парковок, мест, зон, ограничений доступа и статусов.
- Есть автоматизация бронирования и проверка конфликтов интервалов.
- Есть автоназначение места и отдельный endpoint рекомендаций.
- Есть контроль занятости через статусы бронирований и синхронизацию с местами.
- Есть ролевой доступ (admin/owner/tenant/guard/uk).
- Есть контроль доступа транспорта (vehicles/access events + ANPR pipeline).
- Есть аналитика загрузки, аномалий и управленческих рекомендаций.

### 2.2 Возможные формулировки цели
1. Разработка и реализация веб-системы управления парковочными местами частных предприятий с поддержкой ролей, бронирований, контроля занятости и API-ориентированной архитектуры.
2. Разработка интеллектуально-ориентированного модуля подбора парковочных мест на основе ограничений, ролей и скоринговой модели в составе системы управления парковкой.

### 2.3 Задачи работы (привязка к коду)
- Проанализировать предметную область парковочного управления (`docs/DOMAIN_REFERENCE.md`, модели `app/models/*`).
- Спроектировать архитектуру backend/frontend (`app/main.py`, `app/api/v1/router.py`, `frontend/src/app/router/*`).
- Разработать модели БД и миграции (`app/models/*`, `alembic/versions/*`).
- Реализовать API аутентификации и RBAC (`auth.py`, `api/deps.py`).
- Реализовать API парковок/мест/бронирований (`parking.py`, `parking_spots.py`, `bookings.py`).
- Реализовать авто-подбор мест и рекомендации (`services/recommendations.py`, `endpoints/recommendations.py`).
- Реализовать frontend-интерфейс с роль-ориентированной навигацией (`frontend/src/pages/*`, `route-config.ts`).
- Провести тестирование ключевых сценариев (`tests/test_*.py`).

### 2.4 Объект и предмет
- Объект исследования: процессы управления парковочными ресурсами частных предприятий.
- Предмет исследования: методы и программные средства автоматизации бронирования, контроля занятости и алгоритмического подбора парковочных мест.

### 2.5 Научная новизна (аккуратно)
- Корректно заявляемая новизна: интеграция rule-based/scoring-подбора места с explainability-отчетом принятого решения в единой системе бронирования.
- Нельзя заявлять: “полноценная ML-система ИИ”, если используются правила и эвристики без обучения модели.

### 2.6 Практическая значимость
- Для предприятия: автоматизация распределения мест, прозрачность операций, аудит действий, аналитика загрузки.
- Для пользователя парковки: быстрое бронирование, контроль статуса, уведомления, потенциальный auto-assign.
- Для службы охраны/эксплуатации: операционный контроль check-in/check-out/no-show и события доступа.

## 3. Материалы для ГЛАВЫ 1 (теоретические основы)

### 3.1 Задачи системы управления парковками: статус
- Учет парковок — реализовано (`app/models/parking_lot.py`, `endpoints/parking.py`).
- Учет мест — реализовано (`app/models/parking_spot.py`, `endpoints/parking_spots.py`).
- Контроль статусов мест — реализовано (`SpotStatus`, `booking_lifecycle.py`).
- Бронирование/отмена — реализовано (`endpoints/bookings.py`).
- Фильтрация/доступность — реализовано частично (фильтры list API + рекомендации).
- Администрирование — реализовано (`admin_users.py`, audit, notifications).
- RBAC — реализовано (`api/deps.py`, `UserRole`, роуты frontend по ролям).

### 3.2 Интеллектуальность: фактическая оценка
- Автопринятие решения: есть (auto_assign в создании брони).
- Подбор места: есть (`recommend_spots`, `pick_best_spot_for_booking`).
- Проверка конфликтов: есть (overlap по времени + blocking statuses).
- Автообновление статусов: есть (scheduler + lifecycle sync).
- Аналитика: есть (`analytics.py`, `management_recommendations.py`, `anomaly_detection.py`).
- Прогнозирование: присутствуют признаки/документация, уточняется по endpoint’ам аналитики.
- ML: в основном отсутствует в core-рекомендателе; ANPR может использовать внешние модели.

### 3.3 Современные технологии и подходы
- Backend framework: FastAPI.
- ORM: SQLAlchemy async.
- Миграции: Alembic.
- Auth/JWT: `security.py` + auth endpoints.
- Frontend: React/TS/Vite/MUI.
- API client: Axios + React Query.
- Тестирование: pytest-подобные тесты (`tests/*`).
- Документация API/домена: `docs/API_FOR_FRONTEND.md`, `docs/DOMAIN_REFERENCE.md`.

### 3.4 Анализ существующих решений (что брать из проекта)
Для сравнений с коммерческими системами можно использовать критерии:
- наличие ролевой модели;
- качество бронирования и конфликт-менеджмента;
- автоматический подбор;
- аудит и уведомления;
- интеграция с контролем доступа/ANPR;
- аналитика загрузки.

### 3.5 Вывод по главе 1 (тезисы)
- Проект подтверждает ключевые теоретические требования к системам управления парковкой.
- В проекте есть практическое основание для термина “интеллектуальная” на уровне правил/скоринга/рекомендаций.

## 5. Материалы для ГЛАВЫ 3 (разработка)

### 5.1 Выбор технологий
Использовать аргументацию: FastAPI (быстрое API), SQLAlchemy+Alembic (управляемая схема), React+MUI (быстрый интерфейс), React Query (состояние сервера), Axios (REST-клиент).

### 5.2 Реализация backend
Функциональные блоки:
- auth + JWT;
- RBAC через зависимости и проверки роли;
- parking/spot management;
- booking lifecycle (pending/confirmed/active/completed/cancelled/expired/no_show);
- рекомендации (скоринг + explainability);
- аналитика/аномалии/рекомендации управления;
- аудит и уведомления;
- транспортный контур: vehicles/access events/ANPR pipeline.

### 5.3 Реализация frontend
- Маршрутизация с role-based menu (`route-config.ts`).
- Страницы под операции бронирования, анализа, администрирования.
- API-слой и hooks по доменам (`features/*/hooks.ts`, `entities/*/api.ts`).

### 5.4 Интеграция системы
- FE использует общий HTTP-клиент и token storage.
- Передаются JSON-запросы к `/api/v1/*`.
- Сценарий “создание бронирования” реализован end-to-end (форма -> API -> валидация -> запись -> отображение результата).

### 5.5 Вывод по главе 3
Система реализована как единое приложение с завершенным базовым контуром управления парковкой и расширенным функционалом аналитики/рекомендаций.

## 6. Материалы для ГЛАВЫ 4 (исследовательская часть)

### 6.1 Постановка интеллектуальной задачи
Наилучшая задача для диплома: **интеллектуальный выбор оптимального места** с учетом доступности, роли, зон, типа места, зарядки и конфликтов.

Уже есть:
- scoring/ранжирование кандидатов;
- explainability по факторам;
- reject reasons;
- decision report.

Не хватает для “сильной” научной части:
- формального сравнения с baseline;
- набора метрик на реальных данных;
- возможно, статистической валидации.

### 6.2 Алгоритм
Алгоритм уже присутствует в `app/services/recommendations.py`:
- фильтрация кандидатов по ограничениям;
- исключение blocked и interval conflict;
- ролевые ограничения;
- вычисление итогового score как взвешенной суммы факторов;
- сортировка и формирование explainability/decision report.

Псевдокод для диплома можно строить напрямую по этому сервису.

### 6.3 Реализация алгоритма
Текущая интеграция:
- service: `recommendations.py`;
- API: `POST /recommendations/spots`, `POST /recommendations/decision-report`;
- интеграция в бронирование: `bookings.py` (`auto_assign`).
- frontend: компоненты рекомендаций (`features/bookings/components/recommendation-list.tsx`, related panels).

### 6.4 Анализ эффективности (что считать)
Доступные данные: бронирования, статусы, интервалы времени, места, парковки, события доступа.

Рекомендуемые метрики:
- доля успешных бронирований;
- доля конфликтов/отказов;
- загрузка парковки по интервалам;
- доля no_show и cancelled;
- сравнение manual vs auto_assign по времени и успеху;
- равномерность распределения мест.

Ограничения анализа:
- требуется репрезентативный датасет и сценарии нагрузки;
- часть “эффекта” может зависеть от настроек весов.

### 6.5 Вывод по главе 4
Интеллектуальная часть проекта уже реализована как rule/scoring-механизм принятия решения; для усиления диплома необходима экспериментальная оценка по метрикам.

## 11. Риски и ограничения
- Риск “переоценки ИИ”: в рекомендациях rule-based scoring, не полноценное ML.
- Риск недостатка экспериментальной базы: нужны данные и протокол экспериментов.
- Риск по UX: часть сложных экранов (аналитика/операции) требует полевых сценариев usability.
- Риск по документации: нужна консолидация API и архитектурных диаграмм в одном месте.

## 13. Итоговый краткий конспект (1–2 стр.)

Система представляет собой web-платформу управления парковками частных предприятий с backend на FastAPI и frontend на React/TypeScript. Реализованы роли пользователей, управление парковками и местами, создание и сопровождение бронирований, контроль жизненного цикла статусов, уведомления и аудит действий. База данных моделирует ключевые сущности домена: пользователей, парковки, парковочные места, брони, зоны, транспортные средства и события доступа.

API организован по версии `/api/v1` и покрывает как базовые CRUD-процессы, так и доменную логику (конфликты интервалов, правила парковки, операционные статусы, рекомендации). Важной частью проекта является интеллектуально-ориентированный модуль рекомендаций: система ранжирует кандидатов мест по набору факторов и весов, формирует объяснимый отчет и может автоматически назначать место при создании бронирования.

Интеллектуальная составляющая в текущем виде основана на правилах и скоринге, что корректно позиционируется как decision-support, а не как полноформатное машинное обучение. Для усиления исследовательской главы требуется экспериментальная оценка эффективности: сравнение ручного и автоматического выбора по метрикам успешности бронирований, конфликтности, загрузки и эксплуатационных показателей.

Для защиты диплома можно демонстрировать: архитектуру системы, ER-модель, сценарий создания бронирования (manual/auto), экран аналитики, журнал аудита, а также отчет объяснимости рекомендаций.

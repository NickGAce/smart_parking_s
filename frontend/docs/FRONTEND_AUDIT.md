# Frontend Audit — Smart Parking SPA (Design Readiness)

Дата аудита: 2026-04-17
Цель: дать полную картину текущей клиентской части перед редизайном в сторону современного UI.

## 1) Executive summary

Текущее frontend-приложение функционально зрелое для демо и интеграции с backend, но визуально находится на уровне инженерного прототипа.

- **Сильная сторона**: хорошая модульная архитектура и читаемая структура кода.
- **Главный дизайн-риск**: отсутствует полноценная design system (tokens, типографическая шкала, семантические компоненты, единый контент-гайд).
- **Итог для редизайна**: платформа готова к UI-обновлению без переписывания доменной логики, если redesign выполнить через слой `shared/ui` + тему MUI + унификацию page templates.

## 2) Scope и метод

Проверены:
- `frontend/src` (роутинг, layouts, shared/ui, pages, features, widgets)
- конфигурация сборки и линтинга
- существующие фронтенд-документы по архитектуре и ограничениям интеграции.

Проверки выполнены командами:
- `npm run lint`
- `npm run build`

## 3) Текущая зрелость по слоям

### 3.1 Архитектура и масштабируемость — **8.5/10**

- Структура по слоям (`app`, `shared`, `entities`, `features`, `widgets`, `pages`) хорошо подходит для роста и постепенного редизайна.
- API, бизнес-логика и UI в целом разделены корректно.
- Router и role-gating централизованы.

Вывод: архитектурно проект удобен для безопасного рефакторинга интерфейса.

### 3.2 UI consistency — **5.5/10**

- Плюс: есть reusable-компоненты (`PageHeader`, `FiltersToolbar`, `StatusChip`, `PageState` и т.д.).
- Минус: очень много page-level верстки и локальных UI-решений, из-за чего визуальная консистентность зависит от конкретной страницы.
- Минус: mixed language в интерфейсе (русский + английский в заголовках/лейблах).

Вывод: нужно поднимать уровень абстракции UI-компонентов и ввести контентные правила.

### 3.3 Theming & design tokens — **3.5/10**

- Сейчас theme минимальный: фактически только `primary`/`secondary` и светлый режим.
- Не хватает: токенов отступов/радиусов/теней/состояний, типографической системы, semantic palette (success/warning/info/error в контексте продукта), dark mode strategy.

Вывод: главный технический долг для современного дизайна — это отсутствие системного theme foundation.

### 3.4 UX/IA и пользовательские сценарии — **6/10**

- Пользовательские потоки покрыты (auth, booking, notifications, analytics, admin).
- Однако страницы местами выглядят как «операционные формы» без приоритизации контента, выразительной иерархии и явных call-to-action паттернов.

### 3.5 Техническое качество front platform — **7.5/10**

- Lint/build проходят стабильно.
- Есть риск по performance: один крупный production chunk.
- Для редизайна потребуется route-level code-splitting, иначе визуальные улучшения увеличат нагрузку на начальную загрузку.

## 4) Что уже хорошо для будущего дизайна

1. Единый AppShell (sidebar/topbar) — хорошая точка для обновления navigation UX.
2. Централизация API + React Query — UI-рефакторинг не ломает данные.
3. Наличие reusable primitives в `shared/ui` — можно эволюционно перейти к design system.
4. Role-aware routing уже есть — можно сделать role-specific UX-варианты без изменения backend.

## 5) Основные проблемы, мешающие «красивому и современному» UI

### P0 — Нет полноценной design system

Симптомы:
- theme почти пустой;
- вариативность размеров/отступов/стилей задается локально через `sx`;
- нет единой token-стратегии и composable UI kit.

Риски:
- медленный редизайн;
- визуальные расхождения между страницами;
- дорого поддерживать новые фичи.

### P0 — Слабая контентная консистентность

Симптомы:
- смешанные языки в интерфейсе и терминологии;
- неодинаковые названия однотипных действий (`Details`, `Открыть`, `Edit`, `Редактировать`).

Риски:
- ощущение «сырых» экранов даже при красивом визуале;
- снижение UX-доверия в демо.

### P1 — Page composition недостаточно стандартизирована

Симптомы:
- разные страницы по-разному собирают фильтры, таблицы, карточки, заголовки, состояния;
- повторяются похожие layout-паттерны в page-коде.

Риски:
- сложнее внедрять единый дизайн;
- много ручной миграции при редизайне.

### P1 — Performance ceiling перед визуальным апгрейдом

Симптомы:
- build предупреждает о большом чанке.

Риски:
- после добавления richer UI (анимаций, иллюстраций, графики) TTI может ухудшиться.

## 6) Аудит ключевых UX-зон

### 6.1 Navigation shell

Текущее состояние:
- рабочий responsive drawer/topbar;
- базовые breadcrumbs и route titles.

Для redesign:
- усилить brand layer (logo area, hierarchy, account panel);
- унифицировать active/hover/focus states;
- ввести дизайн-паттерн для уведомлений и глобальных быстрых действий.

### 6.2 Data-heavy страницы (parking spots/lots, bookings)

Текущее состояние:
- хороший functional coverage, но визуально перегруженные фильтры/таблицы;
- мало «воздуха» и приоритизации.

Для redesign:
- перейти к модульным screen templates:
  - Header + KPI strip (опционально)
  - Filters panel
  - Primary data view (table/cards)
  - Context drawer/details
- стандартизировать density (comfortable/compact).

### 6.3 Формы и диалоги

Текущее состояние:
- формы функциональны, но не унифицированы как продуктовый form system.

Для redesign:
- общий набор form patterns:
  - field groups,
  - helper/error text,
  - primary/secondary actions,
  - sticky action bar в длинных формах.

### 6.4 Состояния интерфейса

Текущее состояние:
- loading/error/empty покрыты.

Для redesign:
- сделать брендированные пустые состояния,
- систематизировать skeleton/loading UX,
- унифицировать tonal alerts и inline errors.

## 7) План редизайна (рекомендуемый)

### Фаза 1 (1–2 недели): Foundation

- Ввести расширенный theme в MUI:
  - palette tokens,
  - typography scale,
  - spacing/radius/shadow tokens,
  - component variants (Button, Card, Paper, TextField, Table, Chip, Alert).
- Создать UI governance документ:
  - язык интерфейса (рекомендуется RU-only или EN-only в рамках одной сборки),
  - naming/labeling rules,
  - tone of voice.

### Фаза 2 (2–3 недели): Shared UI kit и templates

- Расширить `shared/ui` до семантических компонентов:
  - `PageSection`, `DataPanel`, `FormSection`, `ActionBar`, `MetricCard`, `EmptyStateIllustrated`.
- Ввести 2–3 эталонных page template (data list, details, form-heavy).

### Фаза 3 (2–4 недели): Миграция приоритетных экранов

1. Dashboard + Analytics (визуальный wow-фактор).
2. Parking spots/lots (операционные core-сценарии).
3. Booking flows (критично для демонстрации продукта).
4. Admin screens.

### Фаза 4 (параллельно): Performance + polish

- Route-level lazy loading.
- Оптимизация тяжелых таблиц и графиков.
- Проверка доступности (focus order, контраст, keyboard navigation).

## 8) KPI успеха редизайна

Предлагаемые метрики:
- Time-to-consistent-screen (новая страница в новом стиле): < 1 дня.
- Доля экранов на template-based layout: > 80%.
- Доля inline `sx` со сложной визуальной логикой: снижение минимум на 40%.
- Lighthouse performance/mobile: не хуже baseline до редизайна.
- UX-оценка внутренней демо-команды: +30% к субъективной «современности» интерфейса.

## 9) Приоритетный backlog (дизайн-ready)

### Must-have (P0)
- Расширенный theme + tokens.
- Единый язык и терминология UI.
- Семантические page templates для data/form экранов.

### Should-have (P1)
- Route-level code-splitting.
- Унифицированные form patterns.
- Улучшенные empty/loading states.

### Nice-to-have (P2)
- Dark mode.
- Motion guidelines (микроанимации).
- Библиотека бренд-иллюстраций/иконок для пустых состояний.

## 10) Итог

Frontend уже достаточно структурирован, чтобы провести качественный редизайн без дорогой переработки бизнес-логики.

Главная стратегия: **не переписывать домен**, а внедрить **design system + page templates** и мигрировать экраны по приоритету. Это даст быстрый визуальный скачок и долгосрочную устойчивость интерфейса.

# Smart Parking SPA — Form & Dialog Patterns (RU)

## Цель
Единый стандарт для форм и диалогов: быстрый ввод, предсказуемые действия, понятные тексты на русском языке.

## Ключевые правила

### 1) Структура формы
- Используйте `FormSection` для каждого логического блока полей.
- Заголовок секции должен отвечать на вопрос «что здесь заполняется».
- Дополнительные пояснения размещайте в `helperText` секции или через `InlineFieldHint` у поля.

### 2) Подписи и подсказки
- Подписи полей — на русском, кратко и без технического жаргона.
- Обязательные поля отмечаются `required` (визуальная звездочка MUI).
- Для коротких уточнений используйте `InlineFieldHint`.
- Для ошибок валидации используйте `ValidationMessage`.

### 3) Валидация
- Ошибка должна объяснять, что исправить: формат/диапазон/ограничение.
- Не использовать англоязычные служебные формулировки (`min`, `max`, `spot_number`, и т.д.) в пользовательских сообщениях.
- Серверные ошибки показываются отдельным `Alert` над полями.

### 4) Область действий
- Используйте `FormActions` для пары primary/secondary действий в форме.
- Primary-кнопка — `contained`, справа (или внизу на мобильном).
- Secondary-кнопка — `outlined`/`text`, слева от primary.
- Для длинных форм используйте sticky-область через `FormPageTemplate` + `StickyActionBar`.

### 5) Диалоги
- Для заголовка диалога используйте `DialogHeader` (title + опциональный subtitle).
- Для подтверждений — `ConfirmDialog`.
- Для опасных действий (удаление/безвозвратные операции) — `DestructiveConfirmDialog`.
- В destructive-диалогах обязательно явно указывать необратимость действия.

## Референс-экраны после унификации
- Каталог парковок: создание новой парковки (диалог + форма).
- Реестр парковочных мест: создание, редактирование и удаление (включая destructive confirm).

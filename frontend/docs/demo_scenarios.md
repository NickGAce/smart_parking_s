# Demo сценарии для дипломной защиты

Документ помогает быстро пройти сквозные пользовательские сценарии в предсказуемом порядке.

## Рекомендуемый набор ролей

- **admin** — для показа управленческих и административных экранов.
- **tenant** — для пользовательского пути бронирования.
- **guard** — для оперативных lifecycle-действий.

## Подготовка перед показом

1. Запустить backend и frontend.
2. Проверить, что в системе есть хотя бы:
   - 1 парковка,
   - 3–5 парковочных мест с разными параметрами,
   - 1 активное бронирование (для lifecycle),
   - несколько уведомлений (read/unread),
   - данные для аналитики (или seed).
3. Подготовить учетные записи: `admin`, `tenant`, `guard`.

## Сценарий 1. Tenant: регистрация → бронирование → уведомления

1. Открыть `/register`, создать пользователя.
2. Показать автоматический вход после регистрации.
3. Перейти в **Новая бронь** (`/bookings/new`).
4. Создать бронирование в режиме **manual**.
5. Повторить в режиме **auto** (recommendations + auto_assign).
6. Перейти в **Мои бронирования** (`/my-bookings`) и показать новый статус со статусным бейджем.
7. Перейти в **Уведомления** (`/notifications`) и показать unread/read + mark as read.

## Сценарий 2. Admin: каталоги и управление

1. Открыть **Парковки** (`/parking-lots`): фильтры/сортировка/детали.
2. Открыть **Парковочные места** (`/parking-spots`):
   - показать фильтры,
   - сравнить raw/effective status,
   - показать CRUD-действия (create/edit/delete).
3. Открыть **Управление бронированиями** (`/booking-management`):
   - фильтрация,
   - переход в details,
   - lifecycle кнопки (check-in/check-out/no-show).
4. Открыть **Аналитика** (`/analytics`) и пройти все секции dashboard.
5. Открыть **Пользователи** (`/admin-users`) и показать create + update role.
6. Открыть **Журнал аудита** (`/audit-logs`) и показать trace изменений.

## Сценарий 3. Guard: оперативный workflow

1. Войти под `guard`.
2. Открыть **Управление бронированиями**.
3. Отфильтровать confirmed/active.
4. Выполнить check-in/check-out/no-show на тестовых записях.
5. Показать, что список и детали синхронизируются после действий.

## Что проговорить на демонстрации

- Статусы и переходы управляются backend-правилами lifecycle.
- Frontend показывает пустые состояния, ошибки и успешные действия для операторской прозрачности.
- Ролевая модель ограничивает меню и доступ к защищенным маршрутам.

## Ограничения demo из-за backend

1. Нет `GET /admin/users`: невозможно показать полноценный каталог пользователей.
2. Нет realtime уведомлений: обновление inbox через polling.
3. Нет capabilities endpoint: action-level доступ определяется ролью и frontend-guard'ами.
4. Нет aggregate endpoint для аналитики: dashboard собирается из нескольких запросов.
5. Не везде единый формат ошибок: сообщения частично нормализуются на клиенте.
